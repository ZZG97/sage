# Incidents

Use this directory for concise incident reviews that should influence future development.

## When To Write An Incident Note

Write an incident note when:

- A production or scheduler issue caused repeated failures.
- A Feishu/provider/RSS/task behavior exposed a design weakness.
- A safety rule changed because something went wrong or nearly went wrong.
- The fix is not obvious from the code diff alone.

## Incident Format

```md
# YYYY-MM-DD Short Incident Title

## Impact

What user-visible or operational problem happened.

## Timeline

Important times and observations.

## Root Cause

The specific mechanism, not only the symptom.

## Fix

What changed.

## Follow-Up

What should prevent recurrence.
```
