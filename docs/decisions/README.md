# Architecture Decisions

Use this directory for decisions that explain why Sage's architecture changed.

## When To Write An ADR

Write an ADR when a change:

- Alters module boundaries or dependency direction.
- Introduces a durable runtime model.
- Changes persistent identity, schema ownership, or migration strategy.
- Chooses one provider/session/fallback strategy over another.
- Creates a new execution or exposure policy.

Do not write ADRs for routine implementation details.

## ADR Format

```md
# 0001 Short Decision Title

## Status

Accepted | Superseded | Deprecated | Proposed

## Context

What problem forced the decision.

## Decision

The choice we made.

## Consequences

What becomes easier, what becomes harder, and what follow-up work exists.
```
